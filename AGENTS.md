# Repository Direction

RailWay is now a thesis-oriented full-stack train ticket booking platform.

The current primary application surfaces are:

- `TrainTicketPlatformAPI` - ASP.NET Core Web API backend with EF Core, SQL Server, JWT authentication, services, controllers, migrations, and development seed data.
- `train-ticket-frontend` - Vite + React + TypeScript frontend for passengers and admins.
- `TrainTicketPlatformAPI.Tests` - backend tests.
- `PasswordHasher` - small helper utility for password hash generation.

The old WinForms `TrainTicketApp` has been retired from the solution. Do not reintroduce desktop UI assumptions unless explicitly requested.

# Current Product Scope

The project is targeting a realistic educational train-ticket platform, not a minimal class demo.

Passenger-facing scope includes:

- Public home page with station autocomplete and search.
- Search results for live schedules from the API.
- Class selection and seat-map selection.
- Guest checkout with email and consent.
- Logged-in passenger checkout without the guest login prompt.
- Booking summary, simulated payment, successful ticket confirmation, and expiry/error dialogs.
- My account / my tickets view with refund-oriented ticket management.

Admin-facing scope includes:

- Admin login redirects to the admin dashboard.
- Dashboard uses API-backed operational data where available.
- Train manager for physical trainsets, locomotives, EMUs, carriage ordering, and carriage layout selection.
- Route manager with station search, ordered intermediate stops, autogenerated route code/name, create/edit/delete.
- Schedule manager for assigning trains to routes and operating dates/times.
- Pricing and dynamic fare rules.
- Discount management; discount eligibility is applied by rule, while onboard document verification is outside the app.
- User management.
- Booking management, including admin cancellation/refund reason workflows.
- Revenue reporting using backend booking/revenue report endpoints.

# Architecture Notes

- The React frontend must call the Web API over HTTP. It should not reference backend classes directly.
- Backend controllers should expose DTO/request/response contracts rather than EF entities where practical.
- Admin operations should remain protected by JWT role authorization.
- Public passenger search and station lookup may remain anonymous.
- Booking flows must preserve the selected live trip/train/seat data through the whole flow. Avoid falling back to hardcoded EIP demo values.
- Seat availability is trip/date-specific. Do not treat a physical seat as globally unavailable forever.
- Simulated payments are acceptable for thesis/demo scope, but keep the API shaped like a provider-backed flow using safe test tokens such as `tok_success` and `tok_fail`.
- Preserve booking/payment records as business history. Prefer cancellation/status fields over hard deletes for meaningful records.

# Train And Seat-Map Direction

The train manager should model physical compositions well enough for both admin editing and passenger seat selection.

Supported train concepts include:

- Locomotive-hauled IC/EIC trains with reorderable cars.
- EMU fixed sets whose consist should generally not be reordered.
- ED250 Pendolino fixed sets locked to `Express InterCity Premium`.
- Restaurant/dining cars as consist elements, even when they have no passenger-bookable seats.
- Accessible, family, bicycle, quiet, open-space, and compartment car layouts.

Important current ED250/EIP rule:

- ED250 Pendolino is an `Express InterCity Premium` train.
- Unit 1 is first class.
- Unit 2 is second class with family/compartment and open-space seating.
- Unit 3 is dining with wheelchair accessibility.
- Units 4-6 are second class open-space.
- Unit 7 is a quiet second-class cab unit and is not an accessible car.

Seat-map expectations:

- Wrong-class cars should remain visible in the composition but their seats should be disabled/greyed for the selected class.
- First-class compartment cars have 9 compartments.
- Second-class compartment cars have 10 compartments.
- Compartment cars should show compartments connected by a corridor, with toilets at both ends when appropriate.
- Open-space second-class cars should show a 2+2 arrangement with a center aisle and toilets at both ends when appropriate.
- Open-space cars usually have around 85+ seats; compartment cars usually have around 54+ seats depending on class.

# Database And Seed Data

Development seed data is important for demoability. Keep it realistic enough that a thesis demo can be run without manual SQL setup beyond migrations and connection configuration.

Seed data currently covers:

- Admin and passenger accounts.
- Polish stations/localities and neighboring international stations.
- Routes with ordered stops.
- Trains and consists.
- Schedules.
- Prices, discounts, and demo booking/report data where relevant.

When adding new train layout support, update both:

- Backend seed data / DTO mappings / entities as needed.
- Frontend train manager and passenger seat map rendering.

If seed behavior preserves existing rows, remember that a local database may need old train/carriage/seat rows manually cleared before refreshed seeded visuals appear.

# Development Workflow

- Prefer scoped, thesis-friendly changes over quick placeholders.
- Keep the frontend beginner-readable, but do not under-model important railway concepts.
- Use `rg` for searches.
- Use `apply_patch` for manual edits.
- Do not modify unrelated user changes.
- Build after meaningful backend/frontend changes when feasible.

Useful validation commands:

```powershell
dotnet build
dotnet test
cd train-ticket-frontend
npm run build
```

# Local Configuration

The API uses SQL Server through `ConnectionStrings:DefaultConnection`.

Do not commit real SQL passwords. Prefer ASP.NET Core user secrets for local secrets:

```powershell
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "<local connection string>" --project TrainTicketPlatformAPI
dotnet user-secrets set "SeedData:AdminPassword" "<admin password>" --project TrainTicketPlatformAPI
dotnet user-secrets set "SeedData:PassengerPassword" "<passenger password>" --project TrainTicketPlatformAPI
```

The frontend expects the API at `https://localhost:7246` unless configured otherwise.
