# Repository Direction

RailBook is a thesis-oriented full-stack train ticket booking platform.

The current primary application surfaces are:

- `TrainTicketPlatformAPI` - ASP.NET Core Web API backend with EF Core, SQL Server, JWT authentication, services, controllers, migrations, and development seed data.
- `train-ticket-frontend` - Vite + React + TypeScript frontend for passengers and admins.
- `TrainTicketPlatformAPI.Tests` - backend tests for service and API behavior.
- `PasswordHasher` - small helper utility for password hash generation.

The old WinForms `TrainTicketApp` has been retired from the solution. Do not reintroduce desktop UI assumptions unless explicitly requested.

# Current Product Scope

The project is targeting a realistic educational train-ticket platform, not a minimal class demo.

Passenger-facing scope includes:

- Public home page with station autocomplete and one-way or round-trip search.
- Search results for direct and non-direct itineraries, including up to two transfers across three trains.
- Class selection and segment-aware seat-map selection.
- Booking holds with background expiry.
- Guest checkout with email and consent.
- Logged-in passenger checkout without the guest login prompt.
- Multi-passenger and multi-seat order flow with one payment intent.
- Booking summary, simulated payment, successful ticket confirmation, and expiry/error dialogs.
- Real ticket artifacts: QR payloads, PDF tickets, email delivery status, and combined order ticket flow where available.
- Discount selection and backend discount pricing during checkout.
- Loyalty points through the "Moje IC" program, including earning points and redeeming points during payment.
- Optional dog and large baggage add-on tickets attached to bookings and ticket artifacts.
- Invoice generation and later invoice viewing.
- My account / My tickets view with active tickets, travel history, returned tickets, refunds, ticket PDF download, and current/upcoming trip details.
- Passenger help pages for FAQ/how-to-use, refund policy, invoices, and passenger rights/obligations.

Admin-facing scope includes:

- Admin login redirects to the admin dashboard.
- Dashboard uses API-backed operational data where available.
- Train manager for physical trainsets, locomotives, EMUs, carriage ordering, and carriage layout selection.
- Route manager with station search, ordered intermediate stops, autogenerated route code/name, create/edit/delete.
- Schedule manager for assigning trains to routes and operating dates/times.
- Stop-level timetable details, including route stop arrival/departure offsets, platform, track, and calling patterns.
- Operational disruption tools for delays, cancellation reasons, platform/track changes, and passenger-facing banners.
- Open Railway / PLK import tools for admin-reviewed timetable ingestion into local routes, trains, stops, and trips.
- Pricing and dynamic fare rules.
- Discount management; discount eligibility is applied by rule, while onboard document verification is outside the app.
- User management.
- Booking management, including admin cancellation/refund reason workflows.
- Revenue reporting using backend booking/revenue report endpoints.
- Admin audit logs for successful admin-side create/update/delete actions.

# Architecture Notes

- The React frontend must call the Web API over HTTP. It should not reference backend classes directly.
- Backend controllers should expose DTO/request/response contracts rather than EF entities where practical.
- Admin operations should remain protected by JWT role authorization.
- Public passenger search and station lookup may remain anonymous.
- Booking flows must preserve the selected live trip/train/seat/segment data through the whole flow. Avoid falling back to hardcoded EIP demo values.
- Seat availability is trip/date/segment-specific. Do not treat a physical seat as globally unavailable forever.
- Non-direct itineraries are represented as ordered itinerary segments. Preserve segment identity through search, seat selection, booking order creation, checkout, ticket retrieval, and display.
- Round-trip bookings should preserve outbound and return journey direction through order, payment, ticket, and My tickets flows.
- Simulated payments are acceptable for thesis/demo scope, but keep the API shaped like a provider-backed flow using safe test tokens such as `tok_success` and `tok_fail`.
- Preserve booking/payment/invoice/audit records as business history. Prefer cancellation/status fields over hard deletes for meaningful records.
- Audit logging is currently middleware-backed for successful mutating `/api/admin/...` requests. Avoid logging sensitive request bodies or secrets.
- Open Railway / PLK imports must keep RailBook primary keys local. Store provider identifiers in external ID/source fields and use them for matching/upserts.

# Train And Seat-Map Direction

The train manager should model physical compositions well enough for both admin editing and passenger seat selection.

Supported train concepts include:

- Locomotive-hauled IC/EIC trains with reorderable cars.
- EMU fixed sets whose consist should generally not be reordered.
- ED250 Pendolino fixed sets locked to `Express InterCity Premium`.
- Restaurant/dining cars as consist elements, even when they have no passenger-bookable seats.
- Accessible, family, bicycle, quiet, open-space, and compartment car layouts.

Important current ED250/EIP rule:

- ED250 Pendolino is an `Express InterCity Premium` train.
- Unit 1 is first class.
- Unit 2 is second class with family/compartment and open-space seating.
- Unit 3 is dining with wheelchair accessibility.
- Units 4-6 are second class open-space.
- Unit 7 is a quiet second-class cab unit and is not an accessible car.

Seat-map expectations:

- Wrong-class cars should remain visible in the composition but their seats should be disabled/greyed for the selected class.
- First-class compartment cars have 9 compartments.
- Second-class compartment cars have 10 compartments.
- Compartment cars should show compartments connected by a corridor, with toilets at both ends when appropriate.
- Open-space second-class cars should show a 2+2 arrangement with a center aisle and toilets at both ends when appropriate.
- Open-space cars usually have around 85+ seats; compartment cars usually have around 54+ seats depending on class.
- Segment-aware availability must allow the same seat to be sold on non-overlapping route segments.

# Database And Seed Data

Development seed data is important for demoability. Keep it realistic enough that a thesis demo can be run without manual SQL setup beyond migrations and connection configuration.

The preferred schedule source is now a one-time PLK/Open Railway seed snapshot:

- Use the admin Open Railway importer to preview/import realistic route data into a local database.
- Export the reviewed imported data from `/api/admin/open-railway/seed-snapshot`.
- Save the JSON in `TrainTicketPlatformAPI/App_Data/SeedSnapshots`.
- Normal development seeding reads that JSON locally and should not call the external PLK API during startup.
- The handwritten schedule seed is a fallback/reference. `SeedData:UseHandWrittenDemoSchedules=false` prevents always adding it, while `SeedData:UseHandWrittenDemoScheduleFallback=true` keeps tests/empty snapshot setups usable until real snapshots are saved.

Seed data currently covers:

- Admin and passenger accounts.
- Polish stations/localities and neighboring international stations.
- Routes with ordered stops and timetable offsets.
- Trains and consists.
- Schedules, fares, discounts, and disruption-related fields.
- Demo booking/payment/report/invoice/loyalty data where relevant.

When adding new train layout, booking, ticketing, or admin support, update both:

- Backend seed data / DTO mappings / entities / migrations as needed.
- Frontend admin and passenger rendering as needed.

If seed behavior preserves existing rows, remember that a local database may need old train/carriage/seat rows manually cleared before refreshed seeded visuals appear.

# Open Railway / PLK Import Direction

The Open Railway importer is an admin workflow, not an anonymous passenger feature.

- Configure the approved PLK API key through user secrets at `OpenRailway:ApiKey`; do not commit it.
- Import endpoints live under `/api/admin/open-railway` and require admin authorization.
- The admin frontend page is `/admin/open-railway` and is linked as `Rail data import`.
- Route, train, station, stop, and trip records should be upserted using external identity fields, not by replacing local IDs.
- Use the live importer for curation and snapshot generation, not as a passenger-runtime dependency.
- PLK timetable data does not provide full carriage formation/seat-map data. Imported trains may receive RailBook default consist templates by train category, and admins can refine them later in the train manager.
- Batch imports should remain reviewable: support preview/dry-run before writing and report per-route success or failure.

# Development Workflow

- Prefer scoped, thesis-friendly changes over quick placeholders.
- Keep the frontend beginner-readable, but do not under-model important railway concepts.
- Use `rg` for searches.
- Use `apply_patch` for manual edits.
- Do not modify unrelated user changes.
- Build after meaningful backend/frontend changes when feasible.
- When adding persisted backend fields, create an EF migration and remind the user to run `dotnet ef database update`.
- If changing payment, order, booking, ticket, or refund behavior, run backend tests.

Useful validation commands:

```powershell
dotnet build
dotnet test
cd train-ticket-frontend
npm run build
```

# Local Configuration

The API uses SQL Server through `ConnectionStrings:DefaultConnection`.

Do not commit real SQL passwords or API keys. Prefer ASP.NET Core user secrets for local secrets:

```powershell
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "<local connection string>" --project TrainTicketPlatformAPI
dotnet user-secrets set "SeedData:AdminPassword" "<admin password>" --project TrainTicketPlatformAPI
dotnet user-secrets set "SeedData:PassengerPassword" "<passenger password>" --project TrainTicketPlatformAPI
dotnet user-secrets set "OpenRailway:ApiKey" "<approved PLK API key>" --project TrainTicketPlatformAPI
```

The frontend expects the API at `https://localhost:7246` unless configured otherwise.
